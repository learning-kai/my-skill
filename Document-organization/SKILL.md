# Document Organization

## When to Use
- 用户明确要整理文件、文件夹、目录结构、下载目录、桌面、素材库、项目仓库或混合杂项目录。
- 用户希望统一命名、重组层级、归档旧文件、清理重复文件、拆分大目录或建立更清晰的分类。
- 用户提供了一堆文件清单、目录树、截图或零散描述，希望先输出整理方案，再决定是否执行。
- 用户要求按文档、代码、图片、附件、课程资料、项目阶段、时间或用途来分类。
- 用户需要把混乱目录变成可回溯、可维护、可继续扩展的结构。

## Core Goal
把混乱的文件集合整理成极其清晰、稳定、可解释的目录结构，建立统一的层级、统一的命名和统一的归档方式，并在不丢失上下文的前提下尽量减少命名冲突、重复文件和未来维护成本。

## Organization Rules
- 先识别文件内容和用途，再决定分类方式，不要只按文件名机械分组。
- 先定主目录，再定子目录；先按最稳定的主维度分类，再用次维度细分。
- 优先选择用户最容易理解的维度：项目、主题、时间、类型、来源、阶段、用途。
- 只要一个类别内部存在明显子主题、子阶段、子来源或子类型，就应该建子文件夹，不要硬塞在同一层。
- 如果一个文件夹里仍然混有多种内容，就必须继续拆分，直到同一层里只保留同类内容或同一职责内容。
- 混杂文件夹不能作为长期结构，只能作为临时中转区，整理完成后必须清空或收敛成单一职责。
- 当某个目录下文件数量过多、内容差异明显、命名风格不一致或后续会继续增长时，优先再拆一层子目录。
- 重命名要统一风格，同一目录内尽量保持同一套命名规则，避免同层目录出现多种命名体系。
- 目录名和文件名都要服务于检索：要短、准、稳定、可排序，不要堆砌无意义词语。
- 尽量保留可追溯性：移动文件时说明原位置和目标位置，必要时保留映射表。
- 对重复文件、近似重复文件、压缩包、临时文件和缓存文件要单独标注，不要默认直接删除。
- 如果目录过深或过碎，优先合并重复层级，但不要把语义信息压扁到无法理解。
- 对代码项目要尊重已有项目习惯和构建结构，不要破坏可运行性；对文档和素材库则优先可读性和检索性。
- 当多个分类维度冲突时，优先使用最稳定的主维度，再用次维度做子目录。
- 一个目录如果同时承担了“存放、归档、草稿、输出、备份”多种职责，应拆分为独立子目录，而不是混放。

## Default Workflow
1. 先扫描现状：识别目录结构、文件类型、命名模式、重复项、异常项和高风险文件。
2. 再判断整理目标：是要归档、重命名、去重、扁平化、分层化，还是建立新的索引结构。
3. 设计层级结构：先确定一级分类，再决定是否需要二级、三级子目录，宁可少量明确的层级，也不要把不同东西混在一起。
4. 生成整理方案：给出建议的分类方式、目标目录树、命名规则、子文件夹拆分点和需要特别注意的文件。
5. 如果涉及移动、重命名或删除，先给用户可选方案，而不是直接执行高风险动作。
6. 用户确认后，再输出可执行步骤、批量操作建议或脚本思路，并保持变更可回滚。
7. 完成后给出整理结果摘要：改了什么、为什么这样改、还有哪些文件建议用户复核。

## Plan Before Execution
- 当信息不完整时，先输出简短整理 plan，而不是猜测性地直接改动。
- plan 默认包含：当前问题、建议分类维度、目标结构、重命名规则、子文件夹拆分点、潜在风险、待确认项。
- 如果用户只想要建议，不要强行进入执行阶段。

## Mixed Folder Handling
- 任何同时包含文档、图片、压缩包、安装包、代码片段、临时文件的目录，都应该先拆成不同子文件夹，再继续按用途或主题细分。
- 如果一个子文件夹里还出现明显不同职责的文件，就继续下钻，不要停在“差不多整齐”的层级。
- 目标不是“看起来少一些文件”，而是“每个文件夹只做一件事”。
- 当内容既可按主题分，也可按时间分时，先建主类子文件夹，再在主类下面按时间细分，不要把两种逻辑混写在同一层。
- 如果用户没有明确限制层级深度，宁可多建一层清晰子目录，也不要把不同内容混在同一文件夹。

## Naming Standard
- 文件名优先使用统一、简洁、可排序的格式。
- 日期文件默认使用 `YYYY-MM-DD`，如果需要按月份分组则使用 `YYYY-MM`，按年份归档则使用 `YYYY`。
- 同类文件建议使用一致前缀，例如课程名、项目名、主题名或模块名，避免同层文件名风格混杂。
- 文件名要避免空泛词、重复词和无意义后缀，保留足够的主题信息即可。
- 版本号要统一规则，例如 `v01`、`v02` 或 `v1`、`v2`，同一目录不要混用两套版本写法。
- 终版、草稿、备份、临时件要在命名中明确标识，不要靠猜。
- 如果一个文件名过长，就优先压缩修饰词，保留主体信息和区分信息，不要把所有描述都塞进去。

## Output Patterns
- 混乱下载目录：先按文档、图片、压缩包、安装包、临时文件等粗分，再按日期、来源或用途继续拆分子文件夹。
- 课程/学习资料：按课程名、学期、作业类型、参考资料、提交版分类，必要时再拆分为讲义、作业、实验、考试、补充资料。
- 代码仓库：按模块、功能、配置、测试、构建产物、文档分类，尽量不破坏现有约定，复杂模块下再按职责拆子目录。
- 图片/素材：按项目、主题、用途、拍摄时间或来源分类，并统一命名，素材量大时再按场景、用途、来源拆分。
- 杂项归档：先识别高频主题，再创建少量稳定的大类，避免过度切分；如果杂项持续增长，就建立更细的子目录。

## Hard Constraints
- 不要在没有确认的情况下直接删除文件，尤其是重复项、历史版本和看似无用的目录。
- 不要仅凭文件名猜分类；如果文件名含糊，优先查看上下文或请求补充信息。
- 不要把所有文件都塞进一个“其他”目录，除非它只是临时过渡方案。
- 不要为了追求整齐而破坏项目可用性、引用关系或用户熟悉的结构。
- 不要默认使用过度复杂的目录树；结构应足够简单，让用户下次还能继续维护。
- 不要拒绝建子文件夹：只要类别内部存在明显差异，就应该分层，而不是为了省事全部平铺。
- 不要为了“统一”而把语义不同的文件强行改成同一类名称。
- 不要让目录同时承担多个职责，尤其不要把草稿、成品、归档和备份混放。
- 不要让一个文件夹长期保持“混杂状态”；混杂目录只允许短暂存在于整理过程，不允许作为最终结果。

## If the User Wants Direct Execution
- 先确认范围：哪些目录要整理，哪些文件不能动，是否允许重命名、移动、合并、删除。
- 再给出执行清单：移动路径、重命名映射、保留项、风险项、回滚方式。
- 如果某个目录适合继续下钻，就在执行清单里明确新建哪些子文件夹、每个子文件夹收纳什么内容。
- 如果环境支持脚本化操作，可以生成可审阅的执行步骤或批处理思路，但仍要先确认。

## If the User Wants Only Recommendations
- 直接给出方案、目录树和命名规范，不进入实际改动。
- 同时说明为什么这样分、可能的替代方案，以及各方案的风险差异。
