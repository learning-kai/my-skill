# Task Decomposer

## Overview

从第一性原理拆解任务：先推导目标状态、当前状态、关键差距和必要状态变化，再产出可执行、可验证、可并行调度的任务方案。

不要把“任务拆解”做成模板流水账。输出必须帮助执行，而不是把“调研、设计、开发、测试”这种废话换个排版。

## Sanity Gate

拆解前先审判任务本身。不要忠诚地把错误目标拆得很精细，那只是加速撞墙。

先检查：

- 目标是否自相矛盾？
- 范围是否明显爆炸？
- 约束是否不现实？
- 交付物是否缺失或含混？
- 用户是否把方案、执行、验收混成一团？
- 是否存在更小、更直接的目标表述？

如果任务本身有问题，先指出问题，再给出一个“修正后的任务表述”，然后基于修正版继续拆解。不要因为用户要求“直接拆”就跳过明显的目标冲突；最多减少追问，但不能假装问题不存在。

## 强制第一性原理门槛

任何拆解前都先完成这 7 个推导，不得跳过：

| Step | 必须回答的问题 |
|---|---|
| Outcome | 最终要让什么状态变成真？什么叫完成？ |
| Current State | 现在什么已知、已有、受限、缺失？ |
| Gap | 当前状态和目标状态之间差什么？ |
| Required Transformations | 必须发生哪些状态变化？ |
| Atomic Actions | 每个状态变化需要哪些可执行动作？ |
| Verification | 如何证明动作完成且状态变化成立？ |
| Dependencies | 哪些变化必须先发生，哪些可以并行？ |

裁决规则：

- 保留能直接推动目标状态成立的任务。
- 删除只来自流程惯性、没有明确状态变化的任务。
- 重写动作模糊、无法验收、输入输出不清的任务。
- 合并粒度过细且不能独立验证的任务。
- 继续拆解粒度过粗、仍需要明显决策的任务。

## 颗粒度对齐

默认使用 `G3`。如果用户已经明确要求粗略计划、路线图、命令级步骤或子代理分派，按用户要求调整。

| Level | 用途 | 停止条件 |
|---|---|---|
| G0 目标级 | 只明确目标、边界、成功标准 | 能判断是否值得做 |
| G1 阶段级 | 路线图、里程碑、项目规划 | 每阶段有明确交付物 |
| G2 工作包级 | 团队分工、模块拆分 | 每包能分给一个人或 agent |
| G3 原子任务级 | 默认；Codex 或个人可直接执行 | 动作、输入、输出、依赖、完成标准明确 |
| G4 操作级 | 排障、CI、部署、自动化、文件/命令级执行 | 每步可被命令、检查项或具体文件验证 |

缺少关键偏好时，最多问 3 个问题。优先问会改变方案结构的问题：

1. 拆出来给谁执行：人、Codex、多个子代理，还是团队？
2. 要拆到哪个层级：阶段级、工作包级、原子任务级，还是操作级？
3. 最终要聊天输出、落本地文件、机器可读任务表，还是子代理任务包？

如果用户说“先别问”或问题不影响初版拆解，就基于假设继续，并在最终方案列出假设。

当用户对“细一点/粗一点”表达模糊时，先拆一个代表性分支作为样例，请用户判断“太粗 / 刚好 / 太细”，确认后再完整拆解。

### 方案压测（grill-me）

在**询问用户 / 确认关键决策**环节，默认叠加 `grill-me`（实际流程在 `grilling`）做逐问压测：把目标、边界、取舍、失败模式问清楚，再进入完整拆解。一次只问一个问题，用户答完再问下一个；不要一次甩出长问卷。

安装检测与缺失处理（必须先做）：

1. 检查本机是否已有可用 skill：
   - `skill_view(name='grill-me')` 和/或 `skill_view(name='grilling')` 成功；或
   - 存在 `~/.hermes/skills/grill-me/SKILL.md` 与 `~/.hermes/skills/grilling/SKILL.md`（当前 profile 的 skills 目录）
2. **若未检测到**：
   - 不要静默跳过，也不要擅自安装
   - 用 `clarify` 询问用户是否下载安装，选项示例：
     - `安装 grill-me + grilling`
     - `这次跳过，按普通追问继续`
   - 用户同意安装时执行（非交互）：
     ```bash
     hermes skills install skills-sh/mattpocock/skills/grill-me --yes
     hermes skills install skills-sh/mattpocock/skills/grilling --yes
     ```
   - 安装后再次 `skill_view` 验证；失败则说明错误，并回退到普通最多 3 问
3. **若已安装或刚装成功**：
   - `skill_view(name='grilling')`（或先 `grill-me` 再跟其指示加载 `grilling`）
   - 按 grilling 流程压测本任务的关键决策树（目标/非目标、约束、颗粒度、执行者、风险、验收）
   - 压测完成并达成共识后，再输出最终任务拆解；不要边烤边输出半成品大计划
4. **用户明确拒绝 grill / 说“先别问 / 直接拆”**：
   - 跳过 grill-me，沿用上面的普通追问与假设规则
   - 在最终方案的假设区注明“未做 grill 压测”

触发优先级：用户点名 `grill-me` / `grilling` / “拷问方案”时必做；复杂、模糊、高风险、contest 任务默认建议做；小而清晰的 quick 任务可不做。

## 模式选择

默认只生成拆解方案，不自动执行拆出的任务。只有用户明确要求“开始执行”“按计划实现”“分派 agent 执行”“落地这个计划”时，才进入执行或分派阶段。

| Mode | 触发 | 行为 |
|---|---|---|
| quick | 简单任务、用户只要快速看法 | 聊天输出任务树和下一步 |
| execute | 默认 | 输出第一性原理推导、G3 任务拆解、依赖、验收 |
| dispatch | 用户要求子代理、并行、分包 | 额外生成可分派任务包和并行边界 |
| contest | 复杂、模糊、高风险、大型任务，或用户要求多方向方案 | 启动五个内部 agent 方案竞赛，再融合成一份最终方案 |

不要默认进入 `contest` 处理小任务。多 agent 不是高级感，是成本；只有复杂性值得时才用。

## Contest 模式

`contest` 使用 5 个内部 agent。它们是内部优化机制，不是交付物。

| Agent | 偏置 | 必须产出 |
|---|---|---|
| strategist | 目标、边界、路线、里程碑 | 完整方案 |
| executor | 步骤、输入输出、验收标准 | 完整方案 |
| skeptic | 风险、隐藏依赖、失败模式、目标冲突 | 完整方案 |
| minimalist | 最小可行路径、范围控制、去冗余 | 完整方案 |
| parallelizer | 依赖图、并行边界、子代理分包 | 完整方案 |

执行规则：

1. 先生成统一 brief：任务原文、第一性原理推导、颗粒度、约束、输出要求、假设。
2. 第一轮隔离生成：每个 agent 只读 brief，不读其他 agent 输出。
3. 每个 agent 都必须给完整拆解方案，不能只评论、只挑刺、只列风险。
4. 如果可用 `multi_agent_v1.spawn_agent`，默认继承当前模型；需要最高思考强度且工具支持时设置 `reasoning_effort: "xhigh"`，不要硬编码模型。
5. 如果当前环境不能真正启动 subagent，就在单 agent 内模拟 5 个独立视角，并明确这是降级路径。
6. 必须等待 5 个内部 agent 全部完成后再综合；如果某个 agent 超时、失败或输出空泛，标为低置信，必要时重跑一次。
7. 综合时不是投票。按第一性原理裁决：共识优先，但允许吸收高价值少数意见。
8. 主方案保持稳健；高收益高风险、非必要但可能放大的想法放入“激进备选项”。
9. 默认删除中间 agent 输出。只有用户明确要求 `audit`、`debug`、`保留中间方案` 时才保留。

内部 agent 输出契约：

```markdown
# 目标理解

# 第一性原理推导

# 任务拆解

# 依赖与并行

# 风险与盲点

# 激进建议

# 需要用户确认的问题
```

质量门槛：

- 不能把 agent 数量扩成 6 个以上，除非用户明确要求。
- 不能把五个 agent 的原始输出甩给用户。
- 不能生成一堆中间文件让用户管理。
- 如果某个 agent 方案空泛，标为低置信并少采纳或重跑。
- 明确区分“内部 contest agent”和“最终执行分派 agent”。内部 5 个 agent 只用于优化方案；最终方案只有在用户要求并行执行或子代理分派时，才输出执行分派角色或任务包。
- 输出执行分派时，优先写成 `work packet` 或任务包；需要命名 agent 时，使用“主控 + 不超过 5 个执行 agent”的结构，并说明它们不是内部 contest agent。

## 输出策略

默认少文件：

- 聊天可解决时，直接在聊天中输出。
- 需要落本地时，只保留 `.task-decomposer/<timestamp>-<slug>/final-plan.md`。
- 只有用户要求机器调度、自动化读取或子代理执行时，额外生成 `tasks.yaml`。
- `contest` 的中间 brief、五个 agent 草稿、综合草稿默认只做临时材料，完成后删除。
- 中间材料若必须落盘，放在 `.task-decomposer/<timestamp>-<slug>/.tmp/`。综合完成后删除 `.tmp`；删除失败时，在最终回复里说明残留路径。

`final-plan.md` 固定结构：

```markdown
# 最终任务拆解方案

## 任务理解

## 第一性原理推导

## 颗粒度设定

## 综合说明

## 主方案

## 任务拆解

## 依赖与并行

## 验收标准

## 风险与防线

## 激进备选项

## 需要用户确认的问题

## 下一步建议
```

`tasks.yaml` 只在需要机器可读任务表时生成，使用此最小结构：

```yaml
tasks:
  - id: T001
    title: ""
    type: ""
    priority: P0
    depends_on: []
    parallelizable: false
    input: ""
    output: ""
    done_when: ""
    assigned_to: human_or_codex
    risk: ""
```

## 原子任务标准

一个任务满足以下条件才停止继续拆解：

- 动作明确，不是“思考、优化、处理、研究一下”。
- 输入明确。
- 输出明确。
- 完成标准明确。
- 依赖明确。
- 可单独验证。
- 能在一个短执行周期内完成。
- 不需要执行者再做关键方案决策。

坏任务要重写：

| 坏任务 | 改写方向 |
|---|---|
| 理解需求 | 从用户输入提取目标、非目标、约束、交付物和未知项 |
| 做技术方案 | 列出可选架构、约束匹配度、取舍理由和推荐方案 |
| 优化代码 | 指定性能/可读性/复杂度指标，说明修改范围和验证方式 |
| 测试一下 | 指定测试命令、测试场景、通过标准和失败记录格式 |

## DAG 优先

任务树只用于阅读，执行必须以依赖表或 DAG 为准。不要只输出漂亮层级树。

执行级拆解必须为核心任务提供：

- `id`
- `depends_on`
- `parallelizable`
- `input`
- `output`
- `done_when`

如果任务之间存在共享前置条件、交叉依赖或汇合点，用依赖表表达，不要硬塞成父子树。树适合理解，DAG 才适合执行。

## 高风险计费/路由计划

当最终价格依赖实际命中渠道、供应商、节点或其他运行期资源时，不要只拆“配置、后端、前端、测试”。必须先从代码证据还原“初选路 → 估价 → 预扣 → 重试换路 → 最终结算”的真实时序，并把每次重试发送前的重新估价、追加预留失败禁止发送、最终成功资源快照和金额守恒测试列为 P0。

详细检查表见 `references/retry-aware-billing-plans.md`。

## 从计划切换到执行时的持续推进

当用户确认“按计划实施”“确认运行”或要求继续完成后续任务时，任务拆解阶段结束，进入执行闭环。此时不能只启动一个后台进程、子代理或长测试就回复“正在执行”，更不能把“进程仍存在”当成“任务正在推进”。

执行规则：

1. 将 DAG 的当前关键路径任务写入任务清单，只允许一个明确的 `in_progress` 主任务。
2. 启动后台任务时必须启用完成通知；如果它是当前关键路径上的唯一阻塞项，继续监控到出现可验证进展、完成或明确失败，不能立即以状态播报结束工作。
3. 区分三种状态：
   - **运行中且有进展**：输出、文件、测试阶段或资源指标持续变化。
   - **运行中但阻塞**：进程存在，但长期无输出/无文件变化/停在网络、锁或交互等待。
   - **已完成**：有退出码、测试结果、构建产物或可验证交付物。
4. 长任务超过合理静默窗口时，主动检查等待点、日志、网络、锁和交互提示；发现阻塞后切换镜像、缓存、网络路径或执行方式，并重新验证，而不是等用户追问“为什么停了”。
5. 子代理结果是候选实现而不是完成证明。结果返回后必须由主代理检查 diff、运行测试并验证真实产物。
6. 只有在以下情况才可以停下来等待用户：需要产品决策、涉及高风险/外部副作用确认、缺少不可检索的信息，或所有可用执行路径都已验证失败。
7. 状态回复必须基于证据，使用“已完成 X / 当前阻塞于 Y / 已采取 Z”而不是泛泛的“已经开始、正在处理”。

常见反模式：

- 后台下载依赖卡住，但因为进程仍在就报告“测试正在执行”。
- 分派 agent 后直接结束回合，没有继续处理可并行工作或验证返回结果。
- 计划任务很多，却在完成第一个阶段前只汇报路线图。
- 用户要求“做完再下一个”，却把阶段性启动当作交付。

## 最终自检

交付前检查：

- 是否先审判了任务本身，而不是盲拆？
- 是否先做了第一性原理推导？
- 是否与用户对齐或合理声明了颗粒度假设？
- 是否每个核心任务都有输入、输出、依赖、完成标准？
- 是否删除了模板化废话任务？
- 是否标出了 DAG 依赖、并行边界和串行依赖？
- 是否把激进备选项单独标出，而不是混入主方案？
- 是否默认只给方案，没有擅自执行？
- 是否遵守少文件策略？
- 是否没有把内部 agent 草稿甩给用户？
